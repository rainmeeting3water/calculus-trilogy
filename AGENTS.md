# calculus-trilogy 專案指引

## 適用範圍

本檔適用於整個 `calculus-trilogy` 專案，並補充全域 `AGENTS.md`。如有衝突，以更嚴格或更具體的規則為準。

## 專案定位

- 本專案是純靜態 HTML、CSS、JavaScript 數學互動教材。
- 專案沒有 npm、`package.json`、build step 或外部 CDN。
- CSS 與 JavaScript 直接內嵌於 HTML。
- Canvas 用於呈現數學圖形與互動視覺化。
- 不得擅自導入框架、第三方套件、套件管理器或建置工具。
- 如確實需要新增依賴或改變架構，必須先說明理由、影響與替代方案，並取得明確核准。

## 修改原則

- 修改前先閱讀相關原始檔、導覽關係與既有教學脈絡。
- 優先採取能解決問題的最小幅度修改。
- 保持現有檔案結構、頁面命名、相對連結與學習導覽。
- 不得重寫整個專案，除非使用者明確要求。
- 不得順帶進行無關的重構、格式整理或文案改寫。
- 不得修改 `.nojekyll`、部署說明或其他部署相關設定，除非取得明確核准。
- 保留使用者現有變更，不得擅自還原或覆寫。

## 數學檢查

修改 FTC、定積分、黎曼和或反導函數內容時，必須核對：

- 定義、假設與定義域。
- 所需的連續性或可積性條件。
- 幾何面積、帶符號面積與累積量的區別。
- 積分上下限、方向與符號。
- 函數與反導函數是否正確配對。
- 公式、範例、圖形數值與結論是否一致。
- 測驗選項、正確答案與回饋說明是否一致。

其他數學原則：

- 視覺互動只能支援觀察、直觀理解與猜想，不得將視覺結果本身稱為正式證明。
- 證明概略、直觀說明與正式證明必須清楚區分。
- 若函數可能跨越 x 軸，優先使用「累積函數」、「帶符號面積」或「帶符號累積量」，避免將結果籠統稱為幾何面積。
- 修改互動計算時，需同時核對顯示數值、Canvas 圖形、表格與教學文字。

## 教學與語言

- 學生端文字以自然、清楚的繁體中文為主。
- 英文數學術語以 `BILINGUAL_GLOSSARY.md` 為專案內部基準。
- 雙語內容須符合自然數學語言，不得逐字硬譯。
- 中文與英文內容應表達相同的數學條件與結論。
- 避免用過度神奇化或絕對化的措辭取代數學解釋。
- STEP 導學頁須維持教師提問、學生操作、觀察重點與概念整理的學習節奏。
- 修改完整互動頁時，應檢查對應 STEP 導學頁是否仍然一致；反之亦然。

## 可及性與測試

- Canvas 若承載重要數學資訊，應提供文字摘要、數值說明或資料表作為輔助。
- 不應只依賴顏色區分函數、區域、正負值或正誤狀態。
- 動態測驗、計算結果與狀態回饋建議使用適當的 `aria-live`。
- `input`、`select` 等控制元件應盡量具有清楚且正確關聯的 `label`。
- 互動功能應能以鍵盤操作，並保留清楚的焦點狀態。
- 修改後應檢查窄螢幕、雙語排版、Canvas 顯示與頁面導覽。
- 發布前必須完成 `TESTING.md` 的人工測試清單。
- `TESTING.md` 尚未完成前，不得宣稱本專案已完成正式發布驗收。
- 未實際執行的測試不得聲稱為已通過。

- 所有 Git 與部署操作均須遵守全域 `AGENTS.md`。
- 禁止使用 `git add .`、`git add -A` 或任何萬用 staging。

- staging、commit 與 push 分別需要明確核准，不得視為同一項授權。
- 不得自動 commit、push、建立或修改 remote、切換分支或部署。
- 不得 force-push。
- 未經明確核准，不得建立、連結或發布 GitHub Pages、Netlify 或其他公開站點。

- 未來部署前，必須先確認發布目錄、公開檔案清單及敏感資料掃描結果。
- 發現秘密、學生資料、受限制 IB 材料或其他高風險內容時，必須停止 staging、push 與部署並回報。
